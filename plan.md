# ggmkimbolyric 생성기 개선 계획 (GGM 킴보 스타일)

## 1) 목표
- 현재처럼 문장을 기계적으로 6어절로 잘라서 끊기는 느낌을 줄이고,
- `지지엠 킴보` 특유의 과격하고 과장된 서사/디스 톤을 유지한 채,
- 한 결과(16줄) 안에서 **문장 구조 중복 없음 + 래퍼 중복 없음**을 보장한다.

## 2) 현재 문제 정리
- 문장 단순 분할(`splitByWordCount`) 때문에 의미 단위가 깨짐.
- 완성 문장만 랜덤 추출하면 톤의 흐름(빌드업/클라이맥스)이 약함.
- 줄 길이 제약(6어절 근처)이 기계적으로 적용되어 랩 호흡이 부자연스러움.

## 3) 핵심 방향
- `내용` 자체를 순화/수정하지 않고, **구성 방식**만 개선한다.
- "문장 선택"과 "줄바꿈"을 분리한다.
- 단순 랜덤이 아니라 **스타일 태그 기반 샘플링**으로 결과를 만든다.

## 4) 데이터 계층 재구성
### 4-1. sentences 메타데이터 추가
- 기존 `api/_lyrics-data.js`를 확장하거나 새 파일(`api/_lyrics-meta.js`) 추가:
  - `text`: 원문
  - `tone`: `threat`, `mock`, `absurd`, `street`, `studio` 등
  - `intensity`: 1~5
  - `pattern`: 문장 구조 ID (예: `A-verb-object`, `place-action`, `threat-declare`)
  - `breakHints`: 권장 끊기 위치(단어 인덱스 배열)

### 4-2. 구조 중복 방지 키
- 한 결과 안에서 `pattern`이 중복되면 재추첨.
- 기존 "문장 인덱스 중복 방지"보다 스타일 일관성에 더 유리함.

## 5) 생성 파이프라인 (16줄)
### 5-1. 블록 단위 생성
- 16줄을 4블록(각 4줄)로 구성:
  - Block1: 워밍업(톤 2~3)
  - Block2: 디스 강화(톤 3~4)
  - Block3: 하이라이트(톤 4~5)
  - Block4: 마무리/펀치(톤 3~5)

### 5-2. 각 블록에서 선택 규칙
- 래퍼: 결과 전체에서 중복 금지.
- pattern: 결과 전체에서 중복 금지.
- tone: 블록 목표 intensity 범위에 맞게 샘플링.
- 문장: 원문 그대로 사용(`text` 변경 없음), `[래퍼]`만 치환.

## 6) 줄바꿈 로직 개선
### 6-1. 목표
- "6어절 근처"를 유지하되, 의미 단위를 먼저 보존.

### 6-2. 규칙
1. `breakHints`가 있으면 우선 사용.
2. 없으면 문장부호(쉼표/조사 경계/서술어 직전) 우선 탐색.
3. 그래도 없으면 5~7어절 범위 최적점 선택.
4. 마지막 토막이 1~2어절이면 앞줄에서 재분배.

### 6-3. API 출력 형태
- 지금처럼 `lines` 배열 유지.
- 단, 한 문장에서 2줄 이상으로 분할된 경우도 자연스럽게 이어지도록 순서 보장.

## 7) 토큰/호환성 전략
- `k3`는 새 로직 사용 유지.
- `k1/k2` 레거시는 유지하되, 내부적으로는 동일한 "의미 단위 줄바꿈" 함수 사용하도록 정리.
- 기존 공유 링크가 깨지지 않도록 디코딩 로직은 그대로 둔다.

## 8) 품질 게이트
### 8-1. 하드 규칙
- 결과 16줄 고정.
- 동일 래퍼 이름 중복 0.
- 동일 pattern 중복 0.

### 8-2. 소프트 규칙
- 평균 어절수 5~7 유지.
- 한 줄 최대 어절수 상한(예: 9) 설정.
- 너무 짧은 줄(1~2어절) 비율 제한.

## 9) 구현 순서
1. 메타데이터 스키마 추가 (`_lyrics-meta.js` 또는 `_lyrics-data.js` 확장).
2. `createClusterLyrics` -> `createStructuredLyrics`로 교체.
3. `splitByWordCount`를 `splitByRhythm`으로 교체(의미 단위 우선).
4. 중복 검사(rapper/pattern) 유틸 추가.
5. 샘플 100회 생성 스모크 테스트 스크립트 작성.
6. 실패 케이스(중복/극단적 줄 길이) 재시도 로직 추가.

## 10) 테스트 체크리스트
- 100회 생성 시 중복 규칙 위반 0회.
- 공유 토큰(`k3`) 재호출 시 결과 동일.
- `k1/k2` 링크 해석 정상.
- 프론트 렌더에서 줄바꿈/가독성 문제 없음.

## 11) 리스크와 대응
- 리스크: 메타 태깅 초기 비용 큼.
- 대응: 우선 상위 120문장만 태깅 후 점진 확장.

- 리스크: 톤 과적합(매번 비슷한 결과).
- 대응: tone/pattern 가중치에 랜덤 지터 추가.
